We were running 3-4 trucks where 2 optimized trucks would suffice."

### TASK (15 seconds)
"I needed to build an automated routing system that would allocate the right truck type per shipment cluster and optimize for cost while respecting timing constraints."

### ACTION (90 seconds)
"This was a classic Vehicle Routing Problem (VRP) with capacity and time-window constraints.

**Technical approach:** I built the solver in Python using OR-tools. Inputs: shipment list with origin, destination, dimensions, weight, time window. Outputs: optimized truck assignments with route sequencing.

**Key innovations:**
1. **Truck type optimization**: Algorithm chose between 32ft, 22ft, 19ft trucks based on cube utilization. Earlier, dispatchers always picked 32ft as default.
2. **Hop additions**: Identified routes where adding a 5km hop could combine two partial loads into one full load. Saved 18% in trips on those routes.
3. **Constraints handling**: Built in delivery time windows, vehicle availability windows, and customer preferences.

**Adoption challenge:** Routing managers were skeptical—they'd been doing this for years. I ran a 2-week pilot in Bangalore zone, comparing solver output vs manual routing daily. Solver beat manual by 15-20% in cost on most days. Showed results in weekly ops review. Then rolled out zone by zone over 2 months.

**Integration:** Worked with tech team to integrate solver output into the dispatch system, so it became part of their daily workflow rather than a separate tool."

### RESULT (30 seconds)
"₹1.2 Cr/month savings from the routing solver alone. That's ₹14.4 Cr/year. Truck utilization improved from 65% to 82%. Tool now used across all regions in Flipkart Large. The solver also caught data quality issues—revealed that 8% of shipment dimensions were entered incorrectly, leading to a separate data quality initiative."

### LEARNING
"OR-tools is a fantastic library, but the real work is constraint definition and stakeholder buy-in. Also, automation reveals data quality issues you didn't know existed."

**🎯 Why this is a killer story for AMXL JP:** Automated routing for heavy shipments = directly applicable. VRP knowledge is gold for AMXL planning.

"Saved 39,000 sq. ft. of staging space across the network—~20% reduction. Reduced manpower cost by ₹40 lakh/year (less manual handling). Direct-load percentage went from 8% to 35%. Used as a model for other Delhivery sortation expansions—new centers were designed with smaller staging from the start."

#### 1. Hub-and-Spoke
- **Structure:** Central hub → spokes radiating out
- **Pros:** Lower fixed cost, simpler operations
- **Cons:** Longer average delivery distance, limited surge capacity
- **Best for:** <500 deliveries/day, concentrated geography

- **Pros:** Balanced cost/service, flexible, scalable

# 9. Network Design Math

## Objective Functions for Optimization (Universal Frameworks)

Every network optimization problem has:
1. **Objective function** (what you're optimizing FOR)
2. **Constraints** (what you're optimizing WITHIN)
3. **Decision variables** (what you CONTROL)

### 1. Cost Minimization (Most Common)

```
Minimize: Total_Cost = Fixed_Cost + Variable_Cost

Where:
  Fixed_Cost = Σ (Hub_i × Hub_Fixed_Cost_i) + Σ (Vehicle_j × Vehicle_Fixed_Cost_j)
  Variable_Cost = Σ (Delivery_k × Cost_Per_Delivery_k)
```

**When to use:** "Reduce costs by X%"

**Sub-variants:**
- Min cost per delivery: Total_Cost / Total_Deliveries
- Min cost per distance: Total_Cost / Total_Distance
- Min operating cost (exclude CapEx): Variable only

**Your example:** Flipkart Large CPS reduction (₹112 → ₹99)

### 2. Service Level Maximization

```
Maximize: Service_Score = w1×OnTime% + w2×FirstAttempt% + w3×CSAT

Subject to: Total_Cost ≤ Budget
```

**When to use:** "Improve on-time to 95%"

**Your example:** Maintaining 95%+ on-time during Diwali peak

### 3. Delivery Time Minimization

```
Minimize: Avg_Delivery_Time = Σ (Customer_i × Time_To_Deliver_i) / Total_Customers

Subject to: Cost ≤ Budget, Coverage ≥ Target%
```

**When to use:** "Offer same-day delivery"

**Your example:** Hyperlocal 10-15 min delivery at Flipkart Minutes

### 4. Capacity Maximization

```
Maximize: Total_Capacity = Σ (Hub_i × Max_Throughput_i) + Σ (Partner_j × Daily_Capacity_j)

Subject to: Cost ≤ Budget, Service_Level ≥ Target
```

**When to use:** "Handle peak season"

**Your example:** Diwali 3x volume at Shadowfax

### 5. Revenue/Profit Maximization

```
Maximize: Profit = Σ (Delivery_i × Price_i) - Σ (Delivery_i × Cost_i)

Subject to: Service_Level ≥ Target, Coverage ≥ Target
```

**When to use:** "Get profitable"

### 6. Multi-Objective (Most Realistic)

```
Optimize: f(Cost, ServiceLevel, Capacity, DeliveryTime)

Multi-objective via:
- Weighted sum: w1×Cost + w2×Service + w3×Capacity + w4×Time
- Pareto optimization: Find solutions where improving one requires worsening another
```

**When to use:** "Balance all priorities"

**Three scenarios in interviews:**

**Scenario A: Cost Leader (w1=50%, w2=30%, w3=20%)**
- Min CPS, accept slower (3-5 day), 93% on-time
- Result: ¥4,000/delivery, profitable

**Scenario B: Service Leader (w1=20%, w2=50%, w3=30%)**
- Max experience, higher cost
- Result: ¥5,200/delivery, next-day, 97% on-time

**Scenario C: Balanced (w1=35%, w2=35%, w3=30%)**
- Compromise both
- Result: ¥4,600/delivery, 2-3 day, 95% on-time

### Common Constraints

**Service Level:**
- On-time ≥ 95%, First-attempt ≥ 90%, CSAT ≥ 4.5/5

**Budget:**
- CapEx ≤ $X, OpEx per delivery ≤ $Y, ROI ≥ Z%

**Capacity:**
- Hub ≤ Max_Throughput, Partner ≤ Max_Daily, Vehicle ≤ Max_Stops

**Geographic:**
- Cover all Tier 1, Max distance ≤ X km, At least 1 hub/region

**Operational:**
- 9am-9pm only (Japan), 2-person teams for AMXL, Installation ≤4/team/day

**Regulatory:**
- Noise restrictions, recycling compliance, building access rules

When asked a network design question:

1. **Clarify objective**: "What's the primary goal?"
2. **Define objective function**: "I'd model this as Min(X) subject to (Y)"

# 11. English Communication

## Business English Skills

Japan is non-native English. The recruiter and hiring team will use English, but expect:
- Clear, simple, structured English
- Slow-down considerations

### Verbal Communication Best Practices

#### 1. Speak Slowly and Clearly
- Indian English can be fast for non-native speakers
- Aim for 130-150 words per minute (vs natural 180-200)
- Pause between sentences
- Articulate consonants

#### 2. Structure Your Answers
**P-R-E-P method:**
- **P**oint: State your conclusion first
- **R**eason: Why this conclusion
- **E**xample: Concrete instance
- **P**oint: Restate conclusion

Example: "I believe pilot-first approach is best (Point). It minimizes risk while validating assumptions (Reason). At Shadowfax, we piloted TMS in one hub for 2 weeks before national rollout, catching 15 critical bugs (Example). So pilot-first is my go-to approach (Point)."

#### 3. Avoid Idioms
- ❌ "We hit it out of the park"
- ✅ "We exceeded our targets significantly"

- ❌ "Let's table this"
- ✅ "Let's discuss this later"

- ❌ "Touch base"
- ✅ "Connect" or "meet"

#### 4. Use Specific Numbers
- ❌ "Significant savings"
- ✅ "₹1.2 Cr per month savings, equivalent to $1.4M USD per month"

#### 5. Pause Before Answering
- 3-5 seconds is acceptable
- Shows thoughtfulness
- Avoids ramble

#### 6. Confirm Understanding
- "To make sure I understand, you're asking about X?"
- "Should I focus on the technical aspects or the business case?"

### Written Communication Best Practices


- Executive summary at top (3-5 bullets)
- Key conclusions before details
- Numbers with units and context (₹1.2 Cr/month, 12% reduction, etc.)

#### 3. Be Concise
- Cut adjectives
- Use bullet points for >3 items

- One idea per paragraph
